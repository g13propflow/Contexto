Antes de empezar: necesitas el rol owner (o system_admin) en tu tenant. Todo se configura desde la app; ya no se edita código ni variables globales.

Panorama: las 3 pantallas y cómo se relacionan

Creamos 3 pantallas para administrar Slack por tenant. Ajustes → Slack es donde nacen los canales (los conectas); las otras dos usan esos canales para decidir a dónde va cada mensaje.

🎛️ Ajustes → Slack
/dashboard/settings
Credenciales + switch maestro + conectar canales (General, HITL, …). Aquí nacen los canales del tenant.
▼  los canales conectados quedan disponibles para  ▼
🔔 Notificaciones → Administración
/dashboard/notifications
Ruteo por tipo de notificación: a qué canal va cada tipo + si es "obligatorio".
🔌 Conexiones

Orden natural: 1) conectar canales en Ajustes → 2) rutear por tipo (Notificaciones) y/o por proyecto (Conexiones). Las fases de abajo lo detallan paso a paso 👇

Referencia ¿Cómo funciona el OAuth de Slack? ▸

Conectar un canal = obtener su webhook URL mediante el "baile" OAuth de Slack. Aquí ves el flujo completo y dónde entra cada variable.

  1. El usuario pulsa Conectar en la tarjeta de un canal.
  2. Frontend → GET /integrations/slack/connect?channel_purpose=…. El backend arma la URL de autorización de Slack con el client_id del tenant, scope=incoming-webhook, redirect_uri=SLACK_REDIRECT_URI y un state (tenant:purpose:nonce).
  3. El navegador va a slack.com; el usuario autoriza la app y elige el canal destino.
  4. Slack redirige el navegador a SLACK_REDIRECT_URI?code=…&state=… → es decir, "llama de vuelta" a nuestro backend (endpoint /callback).
  5. /callback intercambia el code (con client_id+client_secret del tenant) en slack.com/api/oauth.v2.access → obtiene el webhook URL y lo guarda en tenant_slack_config.
  6. El backend redirige el navegador a SLACK_FRONTEND_REDIRECT_URL?slack=success.
  7. El usuario aterriza de vuelta en Ajustes → Slack con la tarjeta en "Conectado ✅".

Por qué una exige https y la otra no:

VariableQuién la llama¿https?
SLACK_REDIRECT_URISlack (servidor externo) → tu backendSí — por eso ngrok en local
SLACK_FRONTEND_REDIRECT_URLTu backend → el navegador (ya local)No — http://localhost sirve

¿De dónde sale cada valor?

ValorDe dónde lo sacas
client_id / client_secretDe la Slack App del tenant → api.slack.com/apps → tu app → Basic Information → App Credentials. Se pegan en la UI (Ajustes → Slack), no son env vars.
Ruta /api/v1/integrations/slack/callbackFija, viene del código (router slack_config.py con prefijo /integrations/slack, montado bajo /api/v1). No cambia nunca.
Ruta /dashboard/settingsFija, es la ruta del front (router del SPA — la pantalla con la pestaña Slack).
Dominio del backend (para SLACK_REDIRECT_URI)Local: la URL pública que te da ngrok http 8000. Prod: el dominio real del API — míralo en el VITE_API_BASE_URL de producción, o en la config de Traefik / pregúntalo a DevOps.
Dominio del front (para SLACK_FRONTEND_REDIRECT_URL)Local: http://localhost:5173 (Vite). Prod: el dominio público del frontend.

Armas la variable = dominio (según entorno) + ruta fija. Ej. local: https://ab12.ngrok-free.app + /api/v1/integrations/slack/callback.

Fase 0 Infraestructura del entorno DevOps · una vez ▸

Una sola vez por entorno, la hace DevOps. El resto de las fases dependen de esto.

1

Dar valor a las URLs globales de OAuth

ℹ️ No es desarrollo nuevo: estas 2 variables ya existen en config/settings.py (slack_redirect_uri y slack_frontend_redirect_url) desde antes de esta tarea, y el código ya las usa. Solo están vacías — basta con darles valor en el entorno. No se agrega ni modifica código.

Son globales (dependen del dominio del servicio, no del tenant). Cómo darles valor depende del entorno:

🖥️ En local (Docker) — se editan directo en el .env (el backend lo lee con env_file=".env" y el contenedor lo monta con env_file):

  1. Agrega en app-saas-service/.env:
    • SLACK_REDIRECT_URI=https://<tunel-ngrok>/api/v1/integrations/slack/callback
    • SLACK_FRONTEND_REDIRECT_URL=http://localhost:5173/dashboard/settings
  2. Reinicia el contenedor (docker compose up -d o restart del api).
⚠️ En local Slack exige https → usa un túnel (ngrok); localhost pelado no lo acepta.

☁️ En prod / staging (Azure DevOps) — el .env lo genera el pipeline desde los variable groups, y solo escribe las variables listadas explícitamente. Por eso hacen falta 2 cosas:

  1. Crear SLACK_REDIRECT_URI y SLACK_FRONTEND_REDIRECT_URL en el variable group (variables-ai-agents-*) con el dominio real (https).
  2. Que pipeline/main.yml las escriba al .env — ya quedaron agregadas en los stages DeployDev y DeployQA:
    • SLACK_REDIRECT_URI=$(SLACK_REDIRECT_URI)
    • SLACK_FRONTEND_REDIRECT_URL=$(SLACK_FRONTEND_REDIRECT_URL)
⚠️ Si agregas las líneas al pipeline pero no creas las variables en el variable group, Azure deja el literal $(SLACK_REDIRECT_URI) en el .env (valor basura, inofensivo salvo para el OAuth). Crea ambas en el mismo cambio y reinicia/redeploy.
⚠️ HTTPS obligatorio: Slack exige que la Redirect URL sea https. En producción con dominio real funciona directo; en local haría falta un túnel (ngrok).
💡 Nota: las viejas SLACK_CLIENT_ID/SECRET globales ya no se usan en runtime (quedaron solo para el seed de la migración) y se retiran en la limpieza final (Fase 3, paso 10).
Fase 1 Configuración por tenant Owner ▸

La hace cada owner desde la app. Cada tenant usa su propia Slack App.

1

Obtener la Redirect URL

En la app, ve a Ajustes → Slack → abre "Agregar credenciales". Ahí se muestra la Redirect URL de tu entorno → cópiala con el botón de copiar.

2

Crear la Slack App del tenant

En https://api.slack.com/apps:

  1. Create New App → From scratch → nombre (ej. "PropFlow – MiEmpresa") → elige tu workspace.
  2. OAuth & Permissions → Redirect URLs → pega la Redirect URL del paso 1 → Save.
  3. Incoming Webhooks → actívalo (toggle On).
  4. Basic Information → App Credentials → copia el Client ID y el Client Secret.
3

Guardar las credenciales en PropFlow

En Ajustes → Slack, pega Client ID y Client Secret → Guardar.

Esto guarda las credenciales cifradas por tenant y deja la integración encendida (switch maestro On).

4

Conectar los canales (por propósito)

En Ajustes → Slack, dentro de la sección "Canales", hay una tarjeta por cada propósito. Cada tarjeta se conecta a un canal de Slack que tú eliges.

Tarjeta (propósito)Para qué es
Notificaciones GeneralesCanal fallback: recibe todo lo que no tenga un canal específico. Conéctalo como mínimo.
Control Manual (HITL)Escalaciones y conversaciones que pasan a control manual.
Ventana WhatsAppAvisos de ventana de 24 h por cerrarse.
Resumen diario de actividadResumen diario de asesores.
Supervisión de AsesoresAlertas de desempeño de asesores.

Para conectar un canal:

  1. En la tarjeta del propósito que quieras, pulsa Conectar.
  2. Te redirige a Slack (OAuth). Autoriza la app y elige el canal destino (ej. #ventas, #hitl…).
  3. Slack te regresa a Ajustes → Slack con un mensaje de éxito. La tarjeta pasa a Conectado, mostrando el nombre del canal y la fecha.
  4. Repite para cada propósito que quieras usar (cada uno puede apuntar a un canal distinto).

Una vez conectada, cada tarjeta tiene dos acciones:

  • Probar → envía un mensaje de prueba a ese canal.
  • Desconectar → elimina ese webhook (deja de recibir en ese canal).
💡 Claves: un canal ≠ un propósito fijo — en Slack eliges el canal que quieras para cada propósito. Los propósitos no conectados hacen fallback al canal General. Por eso conectar "General" es el mínimo indispensable.
🛠️ Si Slack dice "Algo ha salido mal al autorizar…": es un problema de la redirect_uri. Revisa que:
  1. El backend tenga SLACK_REDIRECT_URI definida (Fase 0) — si está vacía, el error es inmediato.
  2. Esa URL esté registrada exactamente igual en tu Slack App → OAuth & Permissions → Redirect URLs.
  3. Sea https (Slack no acepta http). En local necesitas un túnel (ngrok) — ver el recuadro de abajo.
🧪 Probando en local: el flujo OAuth exige https + callback alcanzable. Si solo quieres probar el envío de notificaciones, no necesitas "Conectar": se puede sembrar el webhook directo en BD. Para validar el OAuth real en local, levanta ngrok http 8000, pon esa URL https en SLACK_REDIRECT_URI (+ /api/v1/integrations/slack/callback) y regístrala en la Slack App.
5

Verificar el switch maestro

En el mismo panel, confirma que el switch maestro está On.

⚠️ Apagarlo detiene todos los envíos de Slack del tenant al instante (sirve como interruptor de emergencia).
Fase 2 Ruteo y afinamiento Owner · opcional ▸

Opcional: la migración ya dejó valores por defecto idénticos al comportamiento anterior. Usa esto solo para ajustar.

6

Canal y obligatoriedad por tipo de notificación

Ve a Notificaciones → pestaña "Administración" (solo owner/system_admin). Por cada tipo ajusta:

  • Canal Slack: el propósito al que va (vacío = canal General).
  • Obligatorio: fuerza el envío a Slack ignorando la preferencia del usuario.
No es obligatorio tocarlo: el backfill de la migración ya copió los valores que estaban en código. Solo úsalo para cambiar el ruteo.
7

Canal Slack por proyecto (opcional)

Sirve para que las visitas agendadas de un proyecto lleguen a un canal propio, en vez del canal General del tenant.

Cómo asignarlo:

  1. Ve a Conexiones (/dashboard/connections).
  2. Busca la tarjeta "Canal Slack por proyecto" y despliégala (es colapsable — clic en el encabezado).
  3. Verás la lista de proyectos. En el que quieras, pulsa Configurar (o Editar si ya tenía uno).
  4. En el modal, elige del desplegable uno de los canales ya conectados del tenant (los del paso 4) → Guardar.
  5. El proyecto queda marcado con su canal (pill morado #nombre).
🔗 Cómo se relacionan las dos pantallas: el desplegable de este paso muestra solo los canales que están "Conectado" en Ajustes → Slack (los activos del tenant). Los que están "Sin conectar" no aparecen. Por eso el orden es: primero conectas canales en Settings (paso 4), y luego los asignas por proyecto aquí.
Ejemplo práctico:
  • Si en Settings solo conectaste General → en el proyecto solo podrás elegir General.
  • Si conectas también HITL y Ventana WhatsApp → esos 3 aparecerán en el desplegable.
  • Los que dejaste "Sin conectar" → no son opción.
💡 Regla de resolución: al agendar una visita → si el proyecto tiene canal propio, va ahí; si no, cae al canal General; si tampoco hay General, no se envía.

Para quitar el canal de un proyecto: entra a Editar y elige "Usar el canal del tenant" (opción vacía) → Guardar.

Fase 3 Despliegue, verificación y corte de n8n Operativo ▸

Desplegar, confirmar que todo llega y recién entonces retirar Slack de n8n.

·

Desplegar la rama

Push de feature/SCRUM-1278 en ambos repos → PR → merge → deploy a staging/prod.

Correr las migraciones nuevas: copiar alembic/versions/slk0*.py al contenedor y alembic upgrade head (el docker-compose solo monta ./app, por eso el docker cp).

8

Probar de punta a punta

  • Mueve un lead a visita_agendada (desde nuevo/contactado/calificado → agenda cita o cambia estado) → debe llegar el aviso al canal del proyecto (o al general).
  • Prueba un tipo marcado como Obligatorio.
  • Apaga el switch maestro y confirma que no llega nada; vuélvelo a encender.
9

Retirar Slack de n8n (operativo)

En n8n, retira solo la rama/nodo de Slack para visita_agendada.

🚫 No apagues el webhook N8N_LEAD_STATUS_WEBHOOK_URL completo — lo comparten Pipedrive/Zapier/otras automatizaciones. Solo se retira la rama de Slack de visitas.
⚠️ Orden: hazlo tras verificar en un tenant piloto. Mientras ambos (servicio + n8n) están activos hay una ventana de doble envío controlada; el corte de n8n la cierra.
10

Limpieza final (DevOps)

Retirar las variables SLACK_CLIENT_ID / SLACK_CLIENT_SECRET globales del pipeline / variable group — ya no se usan en runtime.

Extra Tenants que YA tenían Slack (migrados) ▸
No parten de cero. La migración les dejó el switch On y sus canales intactos. Solo deben registrar Client ID/Client Secret (Fase 1, paso 3) para poder reconectar o agregar canales a futuro; los canales ya conectados siguen recibiendo sin tocar nada.
